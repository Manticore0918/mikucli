# Codebase Retrieval

mikucli will add Codebase Retrieval as an explicit, local capability rather than silently injecting workspace context into every model request. Users will run `/index` to build or refresh a SQLite-backed Codebase Index under `.mikucli/codebase_index/`, `/search <query>` to search it directly, and agents will use the read-only `search_codebase` tool when task context requires project structure, symbol behavior, implementation locations, or cross-file relationships.

The Codebase Index will be a full rebuild in v1: it indexes selected source, test, documentation, and build/config files; excludes internal state, secrets, generated output, dependencies, binary files, and oversized files; respects `.gitignore` where practical; and stores file/chunk metadata needed for later incremental indexing. `/index` builds into a temporary SQLite database under `.mikucli/codebase_index/tmp/`, validates the result, then replaces the active index so failures keep the previous index intact.

Code chunks for `.py` and `.java` files will use tree-sitter through required `tree-sitter`, `tree-sitter-python`, and `tree-sitter-java` dependencies. Markdown, XML, TOML, and other non-code text files will use language-neutral 2000-character chunks. Embeddings come from local Ollama using `nomic-embed-text` through explicit config defaults (`MIKUCLI_EMBEDDING_PROVIDER=ollama`, `MIKUCLI_EMBEDDING_MODEL=nomic-embed-text`, `MIKUCLI_OLLAMA_BASE_URL=http://localhost:11434`), and indexing fails fast with actionable messages if Ollama or the embedding model is unavailable.

Search will be hybrid: semantic ranking uses cosine similarity over Ollama embeddings, lexical ranking uses SQLite FTS/BM25, and results are merged with reciprocal rank fusion using `1 / (60 + semantic_rank) + 1 / (60 + lexical_rank)` while preserving raw semantic and lexical scores for display. This keeps retrieval local and inspectable while avoiding the prompt bloat and hidden mutation risks of automatic indexing or always-on context injection.
